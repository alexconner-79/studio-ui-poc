# v0.10.12 — Canvas Rendering Overhaul

**Status:** Complete
**Previous:** [v0.10.11](v0.10.11.md) | **Next:** [v0.11.0](v0.11.0.md)

## Summary

Systematic fix to the rendering pipeline so that every property set in the property panel reliably appears on the canvas. The current architecture has three structural gaps: style-like props defined in node schemas are never applied by the renderer, `fontFamily` is missing from the style type system entirely, and text nodes have no Figma-like auto-sizing. Rather than patch individual symptoms, this release fixes the underlying plumbing and introduces a unified props-to-style bridge that covers all 60+ component types in one pass.

Benchmarked against Figma and Plasmic interaction patterns. Figma uses deterministic style application (every property has exactly one code path) and text auto-resize with hug-content defaults. Plasmic uses style isolation at component boundaries with simplified layout abstractions. Studio's renderer currently has a 2000-line switch statement where each node type hand-picks which props to read, leading to silent gaps.

## Design Decisions

**Why a bridge function instead of fixing each renderer case?** The renderer has 60+ cases and growing. Fixing props one-at-a-time in each case is the whack-a-mole pattern — every new node type or schema change can reintroduce the same bug. A centralised `propsToStyle` bridge means style-like props are handled once, regardless of how many node types exist.

**Why default text to hug sizing?** Figma and Plasmic both default text to auto-size (hug content). Studio currently leaves text at default CSS block display, which makes it stretch to 100% of the parent. This is the single biggest reason text boxes "look wrong" on the canvas — the selection box is far wider than the text content.

**Why disable X/Y for in-flow elements?** Figma shows X/Y for every element because every element has absolute coordinates on the canvas. Studio uses a flow-based layout model (like Plasmic), where most elements are positioned by their parent's flexbox. Showing X/Y fields that silently do nothing creates confusion. Better to clearly communicate when X/Y applies.

## Smoke Tests

### Style Pipeline

1. Select a Heading node, change Font Family in the Typography section — the canvas should immediately reflect the font change
2. Select a Text node, change Color to a token value (e.g. `$color.primary`) — the canvas text colour should update
3. Open a project with custom DS tokens, add a Stack with `gap: $spacing.4` — the gap should use the DS token value, not a hardcoded default

### Props-to-Style Bridge

1. Select a Heading, change `color` in the COMPONENT props section — the heading text colour should change on canvas
2. Select a Button, change `textAlign` in COMPONENT props — alignment should update
3. Change the same property in the Typography/Style section — it should override the COMPONENT prop value

### Text Sizing

1. Add a new Heading node — its selection box should hug the text content, not stretch to the artboard width
2. Add a new Text node — same hug behaviour
3. Manually set width to a fixed value — the text box should respect the explicit width and wrap text

### Position Panel

1. Select an in-flow element (no position set) — X/Y fields should be disabled with an explanatory label
2. Set position to Absolute — X/Y fields should enable and allow editing
3. Select a hug-sized element — Width/Height fields should show computed values as greyed placeholders

## Known Limitations

- **Eyedropper tool** — Deferred from v0.10.7. The EyeDropper API has limited browser support (Chrome/Edge only, not Firefox/Safari).
- **Radial gradient visual overlay** — The gradient editor uses numeric centre X/Y inputs rather than a visual overlay on the canvas preview. A canvas overlay would require coordinate translation between the gradient space and the artboard space.
- **`constraints.scale`** — The `scale` constraint type is defined in `NodeStyle` but not implemented in `resolvedStyleToCSS`. This would require measuring the parent's dimensions at render time.
- **`wordSpacing`, `skewX`, `skewY`, `flexShrink`** — Defined in `NodeStyle` and mapped to CSS, but have no UI controls in the property panel. Low priority as these are rarely used in design tools.
