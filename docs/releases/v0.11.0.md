# v0.11.0 -- Advanced Design

**Status:** Planned
**Previous:** [v0.10.5](v0.10.5.md) | **Next:** [v0.11.5](v0.11.5.md)

## Summary

Professional-grade visual design capabilities for Design Mode. Adds multiple fills and strokes per node, a stackable effects system, and a full gradient editor. These are the "designing shadows to perfection" features -- they matter for polished output but are not blocking the core workflow.

Vector editing (boolean operations, pen tool, blend modes) is deferred to v1.3.0 to keep this release focused on the fill/stroke/effects model that most screens actually use.

## Changelog

### 0.11.1 -- Multiple Fills

- [ ] Evolve `NodeStyle.backgroundColor` to support a `fills` array (backwards-compatible -- single `backgroundColor` string still works)
- [ ] Each fill entry: `{ type: "solid" | "linear-gradient" | "radial-gradient" | "image", ...params }`
  - Solid: `{ type: "solid", color: string, opacity: number }`
  - Linear gradient: `{ type: "linear-gradient", angle: number, stops: { color: string, position: number }[] }`
  - Radial gradient: `{ type: "radial-gradient", center: { x: number, y: number }, stops: { color: string, position: number }[] }`
  - Image: `{ type: "image", src: string, size: "cover" | "contain" | "custom", position: string }`
- [ ] Fills stack visually (later entries render on top)
- [ ] Property panel: stackable fill rows with add/remove/reorder, colour or gradient picker per row
- [ ] Compiler: multiple fills map to CSS `background` shorthand with layered values
- [ ] Single `backgroundColor` string maps to a one-entry solid fill internally (backwards-compatible)

**Key files:** `apps/web/lib/studio/types.ts`, `compiler/types.ts`, `apps/web/components/studio/style-sections.tsx`, `compiler/emitters/*.ts`

### 0.11.3 -- Effects Stack

- [ ] Add `effects` array to `NodeStyle`
- [ ] Effect types:
  - Drop shadow: `{ type: "drop-shadow", x, y, blur, spread, color, opacity }`
  - Inner shadow: `{ type: "inner-shadow", x, y, blur, spread, color, opacity }`
  - Layer blur: `{ type: "layer-blur", radius }`

- [ ] Effects stack and are reorderable in the property panel

- [ ] Existing `boxShadow` string still works as a fallback (backwards-compatible)
- [ ] Compiler:
  - Drop shadow / inner shadow → `box-shadow` (multiple comma-separated values)
  - Layer blur → `filter: blur()`
  - Background blur → `backdrop-filter: blur()`

**Key files:** `apps/web/lib/studio/types.ts`, `compiler/types.ts`, `apps/web/components/studio/style-sections.tsx`, `compiler/emitters/*.ts`

### 0.11.4 -- Framework-Aware Style Controls (FRAMEWORK_CONSTRAINTS)

**Decision:** Style controls that don't apply to the project's target framework are hidden rather than shown with a warning. This prevents designers from using properties that will silently fail at compile time.

- [ ] Add `FRAMEWORK_CONSTRAINTS` map to `apps/web/lib/studio/framework-constraints.ts`:
  - Defines which style properties and effect types are supported per framework (Next.js, Vue, Svelte, HTML, Expo)
  - Initially covers: `backdrop-filter`, `filter`, complex `box-shadow` stacks (Expo unsupported), `mix-blend-mode` (Expo unsupported), CSS custom property themes (HTML static mode limited)
- [ ] Property panel and style sections read `FRAMEWORK_CONSTRAINTS` against the current project's framework setting
- [ ] Unsupported controls are hidden (not shown as disabled) -- a small "?" info disclosure at the section header explains: "Some options are hidden because they're not supported by [Framework]. Switch to a web framework to access all options."
- [ ] Compiler gains a `validateStyleForFramework(style, framework)` guard -- even if a style property somehow reaches the emitter, it is filtered rather than producing broken output
- [ ] Multi-framework projects (a project targeting both web and Expo): show the intersection of supported properties only. Framework toggle in toolbar lets designer preview per-platform constraints. *Full multi-platform design support is a v1.2.0+ feature.*

**Why hidden, not disabled:** Showing a disabled control communicates "this exists but you can't use it now." Hiding it communicates "this doesn't apply to your current context," which is more accurate and less confusing.

**Key files:** `apps/web/lib/studio/framework-constraints.ts` (new), `apps/web/components/studio/style-sections.tsx`, `apps/web/components/studio/property-panel.tsx`, `compiler/emitters/*.ts`

## Known Limitations

- **Boolean operations, pen tool, and blend modes deferred** -- *Addressed in v1.3.0 Advanced Vector Tools.*
- **`backdrop-filter` browser support** -- no Firefox Android support as of 2026. *Permanent browser platform constraint -- not fixable by Studio UI. The compiler emits the property correctly; browser support is the vendor's responsibility. The property panel shows a browser compatibility note.*
- **Gradient editor 10 stop maximum** -- more complex gradients should be authored directly in code. *Permanent UX constraint -- 10 stops covers virtually all practical design use cases. Not planned for expansion.*
- **Multiple fills/strokes not token-bound by default** -- fill colours added via the stackable UI are raw hex values, not token references. *Addressed in v1.0.0 stabilisation pass -- the fill/stroke pickers will include a token selector alongside the colour picker, consistent with the existing single-fill token-aware input.*
- **Expo emitter unsupported effects** -- `backdrop-filter`, complex `box-shadow` stacks, and `mix-blend-mode` are not supported in React Native. *Addressed in this release (0.11.4) via `FRAMEWORK_CONSTRAINTS` -- these controls are hidden for Expo projects so the issue never reaches the compiler.*
- **HTML static mode emits only default theme** -- multi-theme CSS custom properties work with a JS theme switcher. No-JS static sites get only the `:root` (default) theme. *Partial permanent constraint for no-JS deployments. The HTML emitter in this release emits all theme blocks with a minimal JS theme toggle script. Pure no-JS mode can only use the default theme by definition.*
- **Component variants Expo emitter** -- `cva()` pattern is web-only. *Addressed in this release alongside FRAMEWORK_CONSTRAINTS -- Expo emitter uses `StyleSheet.create()` with conditional spread for variant logic.*
