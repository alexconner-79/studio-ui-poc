# v0.11.5 -- Real-Time Collaboration

**Status:** Planned
**Previous:** [v0.11.0](v0.11.0.md) | **Next:** [v1.0.0](v1.0.0.md)

## Summary

Wires up the existing collaboration hooks and UI components (built in v0.6.0 but never integrated) into the editor. Two users editing the same screen see each other's presence, cursors, and node selections in real time. Spec changes are broadcast on save so both editors stay in sync.

See [COLLAB_WIRING.md](../COLLAB_WIRING.md) for the detailed integration reference.

## Changelog

### 0.11.5.1 -- Lock-Based Co-Editing Model

**Decision:** Co-editing uses a lock-based model rather than last-write-wins or full CRDT. When a user begins editing a node (selects and starts modifying properties), that node is soft-locked for other users. This prevents silent overwrites without the complexity of operational transforms. CRDT is deferred to a future release if team size and collaboration intensity demands it.

- [ ] When a user selects a node and begins editing (first property change), broadcast a `node-lock` presence event with `{ nodeId, userId, userName }`
- [ ] Other users see the node highlighted with the locking user's colour and a "Editing by [name]" tooltip
- [ ] Locked nodes are not editable by others -- property panel shows read-only state with lock indicator
- [ ] Lock is released automatically when: user deselects, user saves, user is idle for 30 seconds, or user disconnects
- [ ] If a user tries to edit a locked node: show an inline message "Locked by [name] -- wait or ask them to finish"
- [ ] Filesystem / single-user mode: locking is a no-op (never activates)

**Why not CRDT:** Operational transforms and CRDTs solve conflicts at the character level (e.g., two people typing into the same text field simultaneously). For a visual design tool where edits are property changes on discrete nodes (not text), lock-based editing prevents conflicts more simply and is what most design tools (including Figma on free tier) effectively rely on. Full CRDT deferred to v1.2.0 if usage patterns demand it.

**Key files:** `apps/web/lib/studio/realtime.ts`, `apps/web/components/studio/presence-indicators.tsx`, `apps/web/components/studio/property-panel.tsx`

## Test Plan

### Prerequisites
- Supabase configured with Realtime enabled
- Two separate user accounts (create second via `/signup`)
- Dev server running or deployed to Vercel with Supabase env vars

### Tests

**1. Presence Avatars**
1. Open a screen in Chrome as User A
2. Open the same screen URL in an incognito window as User B
3. Verify User A sees User B's avatar in the top bar (and vice versa)
4. Close User B's window -- verify avatar disappears from User A's view within a few seconds

**2. Cursor Tracking**
1. Both windows open on the same screen
2. Move User A's mouse over the canvas
3. Verify User B sees a coloured cursor following User A's position
4. Move User B's mouse -- verify User A sees User B's cursor
5. Verify cursor colours are distinct

**3. Node Selection Highlights**
1. User A selects a Heading node
2. Verify User B sees a coloured outline around that Heading (User A's colour)
3. User A selects a different node -- verify outline moves
4. User A deselects -- verify outline disappears

**4. Spec Broadcasting**
1. User A adds a new Button node and saves
2. Verify User B's canvas updates to show the new Button without refreshing
3. User B edits the Button text and saves
4. Verify User A sees the updated text

**5. Filesystem Mode Guard**
1. Remove Supabase env vars, restart dev server
2. Open editor -- verify it works normally without errors
3. Verify no WebSocket connections are attempted

**6. Disconnect Handling**
1. Both users on the same screen
2. Kill User B's browser tab abruptly
3. Verify User A's presence list updates within 10 seconds

**7. Rapid Selection Changes**
1. User A rapidly clicks 10+ different nodes in quick succession
2. Verify User B's highlights follow without stacking errors
3. Verify no console errors

## Known Limitations

- **Full tree broadcast on save** (no incremental diff) -- for large screens with many nodes, this increases save latency and network usage for all connected users. *Addressed in v1.2.0 with delta diffing via AST diff in the Bidirectional Sync Level 2 release.*
- **Cursor positions are canvas-relative** -- if two users are at different zoom levels, cursor positions appear offset. *Addressed in v1.4.0 UX Refinement Pass -- cursors will be normalised to canvas coordinates accounting for viewport transform.*
- **Supabase Realtime 200 concurrent connection limit** on the free tier. *Permanent constraint on free tier -- resolved by upgrading Supabase plan. Not an architectural issue.*
- **Collaboration requires Supabase** -- filesystem/local mode is always single-user. *Permanent constraint -- by design. Local mode is for solo development; collaboration is a cloud feature.*
- **Conflicting simultaneous edits** -- the lock-based model (0.11.5.1) prevents most conflicts, but race conditions on rapid lock/unlock are possible. *Full CRDT/OT conflict resolution deferred to a future release post-v1.2.0 if usage patterns demand it.*
