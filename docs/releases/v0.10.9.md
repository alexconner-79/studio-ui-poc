# v0.10.9 — Brownfield Canvas Rendering

**Status:** Complete
**Previous:** [v0.10.8](v0.10.8.md) | **Next:** [v0.10.10](v0.10.10.md)

## Summary

Completes the brownfield story. v0.10.0 delivered the first half — a codebase scanner that reads an existing React component library (local path, npm package, or GitHub repo) and extracts component metadata (names, prop types, variants, states) into the DS. That metadata flows correctly: components appear in the palette with the right props and variants.

This release builds the missing runtime layer: a local module server that reads scanned component source files, transpiles them on demand with esbuild, and serves them as CJS modules that the Studio canvas evaluates in-browser via a React-aware require shim. The component registry gains a `"studio:local"` import path that routes through this server, completing the rendering pipeline for all local brownfield component sources.

The canvas also gets a file-change subscription — when the user edits a component file, an SSE stream notifies Studio and the affected `LiveNode` re-fetches and re-renders automatically.

---

## Why This Matters

The brownfield pitch is: *connect your existing component library and design with your real components*. Without canvas rendering, that pitch is hollow — the user still sees boxes with labels, indistinguishable from a Figma frame. With it, the canvas shows their actual Button, their actual Card, and prop changes reflect immediately.

This also closes the shadcn gap properly. The reference rendering approach shipped in v0.10.4 (Studio's own internal shadcn copies) is an acceptable approximation for unmodified components. But users who have customised their shadcn components — changed colours, added variants, restructured the markup — see the wrong thing. v0.10.9 fixes that by loading their actual files. A subtle "ref" badge is shown on studio:shadcn components when no file path is present, with a tooltip explaining that scanning the project will give accurate rendering.

## Changelog

### 0.10.9.1 — Local Module Server [x] Complete

`POST /api/studio/components/bundle` — accepts `{ filePath, exportName }`, reads the file, transpiles TypeScript + JSX with esbuild, returns the CJS bundle as JSON.

- Security: only absolute paths on the local filesystem accepted; rejected in `NODE_ENV=production` / Vercel
- Module caching: transpiled output cached by file path + mtime; cache invalidated on file change (via `DELETE /api/studio/components/bundle?filePath=...`)
- esbuild externals: React, ReactDOM, `react/jsx-runtime`, Next.js, all Radix UI primitives, shadcn utilities (`clsx`, `tailwind-merge`, `class-variance-authority`, `lucide-react`, etc.)
- `DELETE` endpoint for manual cache invalidation

**Key files:** `apps/web/app/api/studio/components/bundle/route.ts`

---

### 0.10.9.2 — Component Registry: `studio:local` Import Path [x] Complete

Extended the component registry to support loading components from the local module server.

- Added `"studio:local"` to `SUPPORTED_WEB_PATHS`
- `getComponent("studio:local", name, filePath)` fetches from `/api/studio/components/bundle`, evaluates the CJS bundle via a `new Function()` require shim, and caches the result per file
- The require shim provides React, ReactDOM, and `react/jsx-runtime` from the same module instance as the Studio canvas (no duplicate React)
- Added `invalidateLocalFile(filePath)` — removes the per-file cache entry; called on file-change events
- `buildLiveProps` now also strips `filePath` and `exportName` from the props passed to the rendered component
- Graceful degradation: if the bundle fails, `LiveNode` falls back to `StructuralInstance`

**Key files:** `apps/web/lib/studio/component-registry.ts`

---

### 0.10.9.3 — DS Schema: File Path on Scanned Components [x] Complete

Scanned components now carry their source file path so the registry knows what to load.

- `ScannedComponent` type gains `filePath?: string` and `exportName?: string`
- `extractComponentsFromSources` now accepts a `recordFilePaths` flag (set to `true` for local scans)
- `scanLocal` passes `recordFilePaths: true` so every component found records the absolute path of its source file and its export name
- `renderer.tsx` reads `props.filePath` from `ExternalComponent` nodes and passes it as the `filePath` prop to `LiveNode`

**Key files:** `apps/web/lib/studio/types.ts`, `apps/web/lib/studio/codebase-scanner.ts`, `apps/web/components/studio/renderer.tsx`

### 0.10.9.4 — Shadcn Proper Rendering [x] Complete

Replaced the reference-only rendering approach with a visible indicator.

- When `importPath === "studio:shadcn"` and no `filePath` is present, a subtle "ref" badge is rendered over the component in design mode
- The badge tooltip explains that scanning the project will give accurate rendering of customised components
- When `filePath` is present (user has scanned their project and the component carries a file path), rendering routes through `studio:local` — the user's actual file — automatically

**Key files:** `apps/web/components/studio/live-node.tsx`

---

### 0.10.9.5 — Auto-Refresh on File Change [x] Complete

When the user edits a component file in their project, affected canvas nodes re-render.

- `GET /api/studio/components/watch?root=<dir>` — SSE endpoint powered by chokidar; emits `component-changed` events with `{ filePath, event }` payloads
- Heartbeat every 15 s to keep the connection alive through proxies
- `LiveNode` subscribes to the SSE stream when `importPath === "studio:local"` using a singleton `subscribeToFileChanges` manager (one `EventSource` per watched root, shared across all nodes in the same root)
- On `component-changed`: calls `invalidateLocalFile`, increments a `reloadKey`, triggering `getComponent` to re-fetch and re-evaluate the fresh bundle
- Stream is only active in local dev mode; the watch endpoint returns 403 in production/Vercel

**Key files:** `apps/web/app/api/studio/components/watch/route.ts`, `apps/web/components/studio/live-node.tsx`

## Design Decisions

**Why esbuild?** Fast (~1–5 ms per file), handles TypeScript + JSX natively, already adjacent in the Next.js ecosystem. Added as an explicit direct dependency (`esbuild@0.27.3`).

**Why CJS format with a require shim?** ESM blob imports (`import(blobUrl)`) cannot resolve bare specifiers (`import { useState } from 'react'`) without an import map, which is complex to maintain. CJS with `new Function('require', ...)` lets us inject pre-resolved modules (React, Radix primitives) from the Studio canvas's own module graph, guaranteeing a single React instance and correct hook behaviour.

**Why not iframe the user's dev server?** An iframe requires the user to have a dev server running, introduces cross-origin complexity, and cannot be used in hosted Studio deployments. The local module server approach works offline and is deployable.

**Why only local mode for auto-refresh?** File watching requires filesystem access, only available when Studio runs locally. Hosted deployments cannot watch the user's filesystem; in hosted mode the user re-scans manually via the brownfield import modal.

## Known Limitations

- **Rendering environment mismatch** — Permanent constraint. Components render inside Studio's CSS context, not the user's app CSS context. Components relying on global CSS from the user's app may look different in canvas vs production.
- **Inter-component dependency bundling** — See BACKLOG.md — Inter-component dependency bundling.
- **Context provider wrapping** — See BACKLOG.md — Context provider wrapping for brownfield components.
- **Path alias resolution** (`@/`, `~/`, etc.) — esbuild does not automatically read the user's `tsconfig.json` for path aliases. Components using `@/lib/utils` or similar aliases will fail to bundle unless those paths happen to resolve from the component file's directory. A future enhancement would pass the user's `tsconfig` paths to esbuild's `alias` option.
