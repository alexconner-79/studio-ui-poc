# v0.10.11 — AI-Ready Codebase

**Status:** Complete
**Previous:** [v0.10.10](v0.10.10.md) | **Next:** [v0.10.12](v0.10.12.md)

## Summary

Makes the handoff between Studio UI, Cursor, and Claude Code seamless. All three tools operate on the same codebase — the bridge between them is the filesystem. This release eliminates friction by adding spec file watching (Studio hot-reloads when external tools edit specs), a Studio CLI, Cursor rules that teach AI agents the project conventions, and a human-readable design change log.

Replaces the original MCP-based approach planned for this release. MCP would have introduced a separate process and protocol layer between the AI and the code — the opposite of how Cursor and Claude Code actually work. Since all three tools already have direct access to the same repo, the solution is to make the existing codebase AI-friendly rather than build an intermediary.

## Design Decisions

**Why not MCP?** Studio UI, Cursor, and Claude Code all operate on the same repo. MCP is designed to bridge systems that can't otherwise communicate — but here, the filesystem IS the shared interface. Adding an MCP server would introduce a separate process and protocol layer between the AI and the code, which is the opposite of how Cursor works (it reads files directly). The solution is to make the existing codebase AI-friendly.

**Why Cursor rules?** Cursor rules (`.cursor/rules/*.mdc`) are the native mechanism for teaching Cursor about project conventions. They persist across sessions, apply automatically based on file globs, and are the standard approach for AI-assisted development in Cursor.

**Why persist the change log to disk?** The Zustand store is client-side and resets on page refresh. Persisting to `.studio/changes.json` means the change log survives refreshes and is directly readable by Cursor/Claude Code as a file — no API call needed.

## Smoke Tests

### Spec File Watching

1. Open a screen in the Studio canvas at `http://localhost:3000/studio`
2. In a separate terminal, edit the corresponding `.screen.json` in `spec/screens/`
3. The canvas should auto-reload with the changes — no manual refresh needed
4. Verify the SSE connection at `/api/studio/specs/watch` in the browser Network tab (EventSource)

### Studio CLI

From the monorepo root:

```bash
pnpm studio list-screens   # should print all screens with routes
pnpm studio tokens          # should dump current design tokens
pnpm studio lint            # should lint all specs without errors
pnpm studio compile         # should compile all screens to generated components
```

### Cursor Rules

Verify four files exist in `.cursor/rules/`:

- `studio-project.mdc`
- `spec-format.mdc`
- `design-tokens.mdc`
- `compiler.mdc`

Confirm that Cursor's AI agent correctly references monorepo layout, CLI commands, and spec format conventions when prompted — this proves the rules are loaded.

### Design Change Log

1. Open the Studio editor and make canvas changes (add a node, move something, change a style)
2. Save the screen
3. Check the API: `curl http://localhost:3000/api/studio/changelog`
4. Or inspect `.studio/changes.json` in the project root
5. Entries should contain human-readable descriptions like "Added Button to Hero Section" or "Styled Heading: fontSize, color"

## Known Limitations

- **Change log granularity** — Style changes group all changed properties into a single entry. A future enhancement could produce per-property entries for very detailed diffing.
- **Spec file watcher local-only** — The SSE endpoint is disabled in production/Vercel. In hosted mode, external spec edits are not auto-detected (the user refreshes manually).
- ~~**CLI runs from monorepo root only**~~ — Fixed. The CLI now walks up parent directories to find `studio.config.json`, so it works from any subdirectory.
