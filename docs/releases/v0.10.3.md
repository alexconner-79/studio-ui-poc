# v0.10.3 — DS & Project Creation Flow

**Status:** Planned
**Previous:** [v0.10.2](v0.10.2.md) | **Next:** [v0.10.5](v0.10.5.md)

## Summary

The relationship between projects and design systems is currently disjointed — a user creates a project, lands in the editor, and only discovers the DS connection exists later via Project Settings. This release fixes the flow so the two concepts are introduced together and the DS is always visible and meaningful in context. The goal is that after this release, no user should be confused about what a DS is or how it relates to their project.

## Changelog

### 0.10.3.1 — Unified Project Creation Flow

Currently: create project → land in editor → discover DS link in Project Settings later.
Target: create project → select or create DS in the same flow.

- [ ] **Project creation modal** — "New Project" on the dashboard opens a modal (not a full page redirect); steps:
  1. Project name + framework (Next.js / Expo)
  2. Design System — three options: **"Attach existing DS"** (dropdown of user's DSs), **"Create new DS"** (opens DS wizard inline), **"Skip for now"** (can link from editor later)
- [ ] If "Create new DS" is selected, the DS wizard opens as a nested step; on completion, the new DS is automatically linked to the project being created
- [ ] If "Attach existing DS" is selected, a searchable dropdown shows all user DSs with platform badge and token count; selecting one links immediately
- [ ] Project creation completes with DS already linked — user lands in the editor with the DS panel populated, not empty

**Key files:** `apps/web/app/studio/page.tsx`, `apps/web/components/studio/onboarding/welcome-modal.tsx`, `apps/web/app/api/studio/projects/route.ts`

### 0.10.3.2 — Dashboard Layout

- [ ] **Projects and Design Systems are visually distinct sections** on the dashboard — currently they share the same grid without clear hierarchy
- [ ] **DS cards show linked projects** — each DS card shows an avatar cluster or count of projects currently using it (e.g. "Used by 3 projects")
- [ ] **Project cards show DS badge** — each project card shows the name and platform badge of its linked DS, or a muted "No design system" label if unlinked

**Key files:** `apps/web/app/studio/page.tsx`, `apps/web/app/api/studio/projects/route.ts`

### 0.10.3.3 — Project Settings DS Panel

The current DS link UI in Project Settings is a plain dropdown with just a name.

- [ ] **DS summary inline** — after selecting a DS, the settings panel shows a compact summary: platform badge, token count by category (e.g. 12 colour, 8 spacing, 6 typography), component count, and last-modified timestamp
- [ ] **Quick actions from Project Settings** — "Open Design System →" link, "View tokens" expandable section showing the first ~10 tokens as colour swatches or value chips
- [ ] **Unlinking confirmation** — unlinking a DS from a project requires a confirmation step; the confirmation message explains what will change ("Token references in your screens will no longer resolve. Node styles using DS tokens will fall back to raw values.")

**Key files:** `apps/web/components/studio/project-settings-panel.tsx`

---

### 0.10.3.4 — Empty State Guidance in Editor

When a project has no DS linked, the editor currently shows nothing to indicate this — the DS panel is empty with no clear next step.

- [ ] **DS panel empty state** — when no DS is linked, the DS panel shows:
  - A brief explanation of what a DS does ("Tokens, typography, and components that apply across all screens in this project")
  - Two clear CTAs: "Create a new design system" and "Attach an existing one"
  - Both CTAs open the same modal as 0.10.3.1 but skip the project name step (project already exists)
- [ ] **Editor banner (optional, dismissible)** — a single-line banner below the top bar for new projects with no DS: "This project has no design system linked. [Add one →]". Dismissed per-project, stored in localStorage.

**Key files:** `apps/web/components/studio/editor-layout.tsx`, `apps/web/components/studio/ds-panel.tsx`

## Design Decisions

**Why a modal, not a page?** Creating a project is a lightweight action — name + framework + DS choice. A full-page wizard creates unnecessary navigation and breaks the user's mental model. The modal approach matches how Figma, Linear, and Notion handle object creation.

**Why inline DS creation in the modal?** The most common new-user scenario is: "I have a new project and no DS yet." Forcing them to create the DS first (on a separate page), then come back to create the project, is a two-trip flow for a conceptually single action. Nesting the DS wizard inline eliminates this.

**Why not force DS selection?** "Skip for now" is retained because some users genuinely want to start laying out a screen before thinking about DS. Forcing DS selection at creation time would be more friction than value for this group. The empty state guidance (0.10.3.4) handles them on their own timeline.

## Known Limitations

- **Nested DS wizard UX complexity** — embedding the full DS wizard (5 steps) inside the project creation modal is technically feasible but may feel heavy for simple cases. If user research shows this creates friction, the alternative is a two-step abbreviated flow ("Quick DS setup": name + pick a boilerplate starter) with a "Full setup →" link that opens the complete DS workspace. *Decision: implement the full nested wizard first; simplify based on feedback.*
- **"No DS" banner for existing projects** — the banner will appear for all existing projects that have no DS linked, which may be unexpected for users who created projects before this release. *Mitigated by making the banner dismissible per-project.*
