# v0.10.8 — Component Editability

**Status:** Complete
**Previous:** [v0.10.7](v0.10.7.md) | **Next:** [v0.10.9](v0.10.9.md)

## Summary

Components land on the canvas and render correctly after v0.10.4–v0.10.5, but editing them is still a developer experience, not a designer experience. The properties panel exposes raw prop names (`children`, `variant`, `asChild`), hides most of the useful controls, and gives no indication of what a component actually does or what can be changed.

This release makes every component in the Studio boilerplate library fully editable by a designer. Each component gets a curated property panel: designer-friendly labels, all meaningful variants surfaced as a visual control, all states represented, and only the props a designer would ever touch visible by default. Developer-only props (`asChild`, `ref`, `className`) are hidden unless the user explicitly expands an "Advanced" section.

The result is that a designer can drop any component onto the canvas, see it labelled clearly, and immediately understand what can be changed — without needing to read component documentation or ask a developer.

---

## Why This Matters

The component library is only as useful as the editing experience on top of it. A Button with five variants that are only accessible by knowing to type `"outline"` into a raw prop field is no better than a static image. This release is what transforms the component palette from a developer handoff tool into an actual design tool.

## Changelog

### 0.10.8.1 — Component Property Schema

Extend the component definition schema to carry designer-facing metadata for every prop. This schema drives the property panel rendering automatically — no per-component UI code required.

Each prop definition gains:

```ts
type PropDef = {
  name: string;              // internal prop name (e.g. "variant")
  label: string;             // designer-facing label (e.g. "Style")
  type: "string" | "boolean" | "number" | "number[]" | "select" | "color" | "node" | "enum";
  options?: { value: string; label: string; preview?: string }[]; // for select/enum
  default?: unknown;
  description?: string;      // shown as tooltip on hover
  group?: string;            // panel section: "Content" | "Style" | "Layout" | "Behaviour" | "Advanced"
  hidden?: boolean;          // hidden by default; shown in "Advanced" section
};
```

- [x] Extended `PropDef` with `group?: "Content"|"Style"|"Layout"|"Behaviour"|"Advanced"`, `description?: string`, `hidden?: boolean`
- [x] Button: `label` (Content), `variant` (Style, shadcn-aligned), `size` (Style), `disabled` (Behaviour), `asChild` (hidden)
- [x] Badge: `text`→"Label" (Content), `variant`→"Style" (Style)
- [x] Input: `label` (Content), `placeholder` (Content), `type` (Style), `disabled`/`readOnly` (Behaviour)
- [x] Textarea, Label, Skeleton: fully grouped and described
- [x] All form components (Checkbox, Switch, RadioGroup, Select, Slider) + Toggle added: grouped with Behaviour for state/disabled
- [x] All overlay components (Dialog, AlertDialog, Drawer, Sheet, Popover, Tooltip, DropdownMenu): grouped with side/position in Style

## Design Decisions

**Why group props into Content / Style / Layout / Behaviour / Advanced?** Designers think in terms of what a control does, not what a prop is called. Grouping into these categories matches mental models: Content = what it says, Style = how it looks, Layout = how it sits, Behaviour = how it acts, Advanced = developer overrides. The grouping is also consistent across all components, so experienced users always know where to look.

**Why hide `className`, `asChild`, and `ref` by default?** These are escape hatches for developers, not controls for designers. Surfacing them alongside `variant` and `size` creates noise and signals "this is a developer tool." Hiding them in Advanced keeps the default view clean. Power users who need them know to look in Advanced.

**Why show variant thumbnails in the palette?** Knowing that Button has `outline` and `ghost` variants is useless if you have to drop the button, find the variant prop, type the value, and see the result. Variant thumbnails let designers visually choose what they want before placing it.
