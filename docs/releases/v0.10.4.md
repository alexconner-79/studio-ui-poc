# v0.10.4 — Component Instances

**Status:** Complete
**Previous:** [v0.10.3](v0.10.3.md) | **Next:** [v0.10.5](v0.10.5.md)

## Summary

When a user drops a component from the DS palette onto the canvas, they should get a live, editable instance of that component — not a placeholder. A Card component becomes a Card on the canvas with its default content areas (title, body, footer) immediately editable. Props like size and variant are controlled from the property panel. The whole thing compiles to `<Card size="default">...content...</Card>`.

This was always the intended model. The groundwork exists — components have variants, states, and typed props — but the instance model (default template, slot regions, editable children) was never built.

---

## Mental Model

A component instance is a node tree, not a leaf:

- The **root node** represents the component itself, carrying its typed props (size, variant, bordered, etc.)
- Its **children** represent the component's content areas (slots) — editable regions on the canvas
- Those children are **real nodes**: Heading, Text, Image, Stack, etc. — fully selectable, moveable, and styleable
- The **property panel** shows both the component's typed props and the slot structure
- **Compilation** maps typed props to JSX attributes and slot children to JSX children or named prop slots

This is how React components actually work. Studio reflects that model in the canvas rather than inventing a different one.

## Changelog

### 0.10.4.1 — Component Instance Node Type

Introduce `ComponentInstance` as the canonical node type for placed component instances. Replaces `Frame + dsComponentName` (greenfield) and unifies with `ExternalComponent` (brownfield) into a single model.

- [ ] Add `ComponentInstance` to `NodeType` in `types.ts`
- [ ] `ComponentInstance` node shape:
  ```json
  {
    "id": "card-1",
    "type": "ComponentInstance",
    "name": "Card",
    "props": {
      "componentName": "Card",
      "importPath": "antd",
      "size": "default",
      "bordered": true
    },
    "children": [
      { "id": "card-1-title", "type": "Heading", "props": { "text": "Card Title", "level": 4 }, "children": [] },
      { "id": "card-1-body", "type": "Text", "props": { "text": "Card content goes here." }, "children": [] }
    ],
    "style": {}
  }
  ```
- [ ] `importPath` is optional — present for package-based components, absent for custom greenfield components
- [ ] Typed props (size, bordered, etc.) sit alongside `componentName` and `importPath` directly in `props`
- [ ] `ExternalComponent` and `Frame + dsComponentName` become legacy aliases — existing specs continue to work via renderer compat; new placements always create `ComponentInstance`

**Key files:** `apps/web/lib/studio/types.ts`

### 0.10.4.2 — Default Templates

Each component definition carries a `defaultTemplate` — the node tree instantiated when the component is dropped onto the canvas.

- [ ] Add `defaultTemplate` field to the component definition type:
  ```ts
  type ComponentDefinition = {
    name: string;
    description?: string;
    category?: string;
    variants?: VariantDef[];
    states?: string[];
    props?: PropDef[];
    importPath?: string;
    defaultTemplate?: Node;   // the tree to instantiate on drop
  }
  ```
- [ ] If `defaultTemplate` is absent, the component drops as an empty `ComponentInstance` node (no children) — valid for simple leaf components like Button
- [ ] Default templates for all components in the package-based starters are defined in the starter JSON files (see 0.10.4.4)
- [ ] Default templates for custom greenfield components are authored in the DS workspace (see 0.10.4.5)

**Key files:** `apps/web/lib/studio/types.ts`, `apps/web/lib/studio/store.ts`

### 0.10.4.3 — Canvas Rendering and Interaction

`ComponentInstance` nodes render their children directly on the canvas. The component boundary is indicated by a subtle, unobtrusive visual treatment — not a dashed placeholder.

- [ ] **Renderer** — `ComponentInstance` renders as a container whose children are rendered normally; the component boundary shown as a thin `2px solid` accent-coloured outline when hovered or selected (not dashed, not always visible)
- [ ] **Component label** — a small chip showing the component name appears above the top-left corner of the boundary when the node is selected or hovered; same visual treatment as the existing artboard label
- [ ] **Children are independently selectable** — clicking a child node selects it, not the parent instance; clicking the boundary or label selects the instance root
- [ ] **Drop targets** — slot regions accept dropped nodes from the palette; the drop indicator shows inside the correct slot
- [ ] **Instance root in Layers panel** — shown with a component icon (⬡) to distinguish from plain Frame/Stack nodes; children shown nested beneath it
- [ ] **Deselect on canvas click** — works the same as all other nodes

**Key files:** `apps/web/components/studio/renderer.tsx`, `apps/web/components/studio/editor-canvas.tsx`, `apps/web/components/studio/node-tree.tsx`

### 0.10.4.4 — Starter Default Templates

Default templates for all components in the four package-based starters (Ant Design, Material 3 Web, Material 3 Native, iOS HIG).

Templates are defined directly in the starter JSON files alongside the existing component metadata:

```json
"Card": {
  "name": "Card",
  "variants": [...],
  "props": [...],
  "importPath": "antd",
  "defaultTemplate": {
    "type": "ComponentInstance",
    "props": { "componentName": "Card", "importPath": "antd", "size": "default", "bordered": true },
    "children": [
      { "type": "Heading", "props": { "text": "Card Title", "level": 4 }, "children": [] },
      { "type": "Text", "props": { "text": "Card content." }, "children": [] }
    ]
  }
}
```

- [ ] Default templates added to all components in `ant-design.tokens.json`
- [ ] Default templates added to all components in `material3-web.tokens.json`
- [ ] Default templates added to all components in `material3-native.tokens.json`
- [ ] Default templates added to all components in `ios-hig.tokens.json`
- [ ] Leaf components (Button, Badge, Tag, Divider, etc.) get `defaultTemplate: null` — they drop as empty `ComponentInstance` nodes; their content is controlled entirely by typed props
- [ ] Container components (Card, Modal, Drawer, List, Table, Form, etc.) get templates with appropriate child scaffolding

**Key files:** `spec/ds-starters/*.tokens.json`

### 0.10.4.5 — Template Authoring in DS Workspace

Custom greenfield components (those without an `importPath`) need a way to define their default template. The DS workspace Components tab gains a mini canvas for authoring the template.

- [ ] Each component in the Components tab has an "Edit template" action
- [ ] Opens a simplified canvas panel (same node palette as the main editor, same drag-and-drop) showing the component's current template
- [ ] Changes to the template are saved to the component definition in the DS
- [ ] The mini canvas uses the DS tokens for rendering (colours, spacing, typography apply immediately)
- [ ] If no template exists, the panel shows an empty canvas with the prompt "Drag nodes here to define the default structure of this component"
- [ ] Template is stored as a `defaultTemplate` node tree in `design_systems.components` JSONB

**Key files:** `apps/web/components/studio/ds-workspace/ds-components-tab.tsx`

### 0.10.4.6 — Property Panel for Instances

The property panel when a `ComponentInstance` is selected shows typed props and slot management.

- [ ] **Typed props section** — shows all props from the component definition; renders appropriate input per prop type:
  - `string` → text input
  - `boolean` → toggle
  - `number` → number input
  - enum (variant) → chip group
  - `node` / `ReactNode` → slot indicator with "+" to add a child node
- [ ] **Variant selector** — chip group for components with variants (existing behaviour, preserved)
- [ ] **"Reset to default"** — restores children to the `defaultTemplate` state; confirmation required if children have been edited
- [ ] **Detach instance** — converts the `ComponentInstance` node to a plain `Frame` with the same children; removes the component binding; irreversible (confirm dialog)

**Key files:** `apps/web/components/studio/property-panel.tsx`

---

### 0.10.4.7 — Compiler

`ComponentInstance` nodes compile to the correct JSX with typed props and slot children.

- [ ] **Next.js emitter** — emits `import { Name } from 'importPath'` + `<Name typedProp="value">...children...</Name>`
- [ ] **Expo emitter** — same pattern for React Native components
- [ ] **No `importPath`** (custom greenfield) — emits `import { Name } from '@/components/[name]'` (local import convention); warns if the component file does not exist
- [ ] **Named slots** — if a prop is typed `node` and the child node carries a `slotName` prop, it compiles as a named JSX prop: `footer={<div>...</div>}` rather than a child
- [ ] **Legacy compat** — `ExternalComponent` and `Frame + dsComponentName` nodes compile via the same logic as `ComponentInstance`; the emitter normalises them before emitting

**Key files:** `compiler/emitters/nextjs.ts`, `compiler/emitters/expo.ts`

## Design Decisions

**Why `ComponentInstance` instead of extending `Frame`?** `Frame` is a layout primitive. Overloading it with component semantics (import paths, typed props, slot regions) makes the type ambiguous and the compiler logic fragile. A dedicated type is clear, explicit, and forwards-compatible with more complex component features (controlled state, event handlers, data binding) planned for later releases.

**Why children as real nodes rather than prop strings?** Because this is a code-first tool. React components receive children as JSX — not as string prop values. The canvas should reflect that. A Card's title is a node you can style, not a text field in the property panel.

**Why default templates in the starter JSON?** Because the starter is the contract for what a library provides. If you pick Ant Design, you get Ant Design's Card with a reasonable default structure. Keeping templates in the starter file means they can be versioned, inspected, and updated alongside the token set.

## Known Limitations

- **Named prop slots** — Addressed in v0.10.8 Component Editability. Slot management UI (drag children to named slots) is part of that release. See BACKLOG.md — Named prop slot management UI.
- **Template authoring mini-canvas undo/redo isolation** — Permanent constraint. The template editor operates on a separate node tree; undo/redo is local to it and does not interact with the main editor history. By design.
- **Custom component local imports** — Addressed in v0.10.9 Brownfield Canvas Rendering. DS export generates the required component file.
- **Theming for antd/MUI reference components** — Addressed in v0.10.5 via CSS variable injection for shadcn. Ant Design ConfigProvider and MUI ThemeProvider mapping are addressed in v0.10.9 Brownfield Canvas Rendering.
- **Brownfield local components render as structural outlines** — Addressed in v0.10.9 Brownfield Canvas Rendering.

### 0.10.4.8 — Live Component Rendering (Partial — bugs present)

Beyond the spec's structural instance model, an "always-live" rendering approach was implemented: when a `ComponentInstance` has a known `importPath` (e.g. `"antd"`, `"@mui/material"`), the canvas renders the actual npm library component rather than a structural outline.

**Files created:**

- `apps/web/lib/studio/component-registry.ts` — central registry for dynamically loading supported component libraries. Defines `SUPPORTED_WEB_PATHS` (`antd`, `@mui/material`) and `NATIVE_PATHS` (`react-native-paper`, `@react-native-community/ios-hig`). Exports `getComponent` (cached dynamic import), `buildLiveProps` (merges `selectedVariants` + `componentProps`, strips internal keys).
- `apps/web/components/studio/live-node.tsx` — client component that dynamically loads and renders the real library component. Contains `LiveNode` (live render), `StructuralInstance` (fallback for unknown/custom importPaths), `NativeFallbackChip` (badge for native-only libraries).

**Files modified:**

- `apps/web/components/studio/renderer.tsx` — `ComponentInstance` and `ExternalComponent` cases updated to branch on `importPath`: native → `NativeFallbackChip`, supported web → `LiveNode`, unknown/absent → `StructuralInstance`.

**Critical bug — `importPath` never reaches the node:**

The live rendering path is gated on `props.importPath` being set. However `importPath` is never stored on individual component definitions — `packageName` lives at the top level of the tokens.json file and is never propagated to each component when the starter is applied or when `dsComponents` loads into the store. As a result, every `ComponentInstance` node dropped from the palette has no `importPath` and every render falls back to `StructuralInstance`. This is fixed in the v0.10.4 completion pass.

**Critical bug — dot-notation compiler crash:**

Sub-components such as `List.Item`, `Menu.Item`, and `Form.Item` cause the Next.js and Expo emitters to generate `import { List.Item } from "antd"` — invalid JavaScript. The import resolver in both `compiler/emitters/nextjs.ts` and `compiler/emitters/expo.ts` does not strip the sub-component suffix when building named import lists. Fixed in the v0.10.4 completion pass.

### 0.10.4.9 — Shadcn Reference Rendering

shadcn/ui has no npm package. Components are source files in the user's own project. Since Studio cannot load the user's local files at runtime (see v0.10.6), a reference rendering approach is used:

- A virtual import path `"studio:shadcn"` is assigned to the shadcn starter.
- The component registry maps `"studio:shadcn"` to Studio's own internal copies in `apps/web/components/ui/`.
- All 44 shadcn components from the starter are installed into `apps/web/components/ui/` via the shadcn CLI.
- The canvas renders Studio's standard shadcn copies as a reference — visually correct for unmodified components; does not reflect user customisations to their own `components/ui/` files.
- This is documented as a known approximation. The compiled output is unaffected and always generates the correct `import { X } from "@/components/ui/x"` pointing to the user's own files.
