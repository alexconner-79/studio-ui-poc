# v0.10.0 -- Design System Authoring

**Status:** Complete (Path 1 Greenfield + Path 2 Brownfield)
**Previous:** [v0.9.8](v0.9.8.md) | **Next:** [v0.10.1](v0.10.1.md)

## Summary

The design system IS the factory. This release makes Studio UI the best tool for building and connecting design systems for code-first teams. There are two equal paths:

- **Path 1 -- Greenfield**: spin up a new DS fast from a boilerplate or the creation wizard. For new projects that don't yet have an existing component library.
- **Path 2 -- Brownfield**: connect an existing React component library so Studio uses your real components, real tokens, and real TypeScript prop types. For code-first teams who already have a `packages/ui`, an npm package, or a shadcn/Radix + Tailwind setup.

Path 2 is the primary story for the audience building close to code. When the MCP bridge (v0.10.5) ships immediately after, the AI agent in Cursor or Claude Code already understands the real component API -- because Studio does.

### Path 2 -- Brownfield (Connect Existing Codebase)

The primary path for code-first teams. If your design system already lives in your codebase -- as a local `packages/ui`, a published npm package, shadcn/Radix + Tailwind, or any React component library -- Studio UI connects to it rather than recreating it. The palette populates with your real components, the property panel shows your actual prop interfaces, and the compiler outputs the correct import statements.

#### 0.10.5 -- Codebase Component Library Import

Three entry points for connecting an existing library:

- **Local path**: point at a directory (e.g., `./packages/ui/src/components`) -- Studio scans it server-side
- **npm package**: enter a package name (e.g., `@company/design-system`) -- Studio installs it in a temp context and introspects exports
- **GitHub repo**: enter a repo URL -- Studio uses the GitHub API (with the connection from v0.9.8.2) to scan source files

**What Studio extracts from the scan:**
- [x] Component names and named exports (PascalCase exports = components)
- [x] TypeScript prop interfaces via ts-morph AST (regex fallback if unavailable)
- [x] Variant unions from prop types (e.g., `variant: "primary" | "secondary"` auto-populates variant list)
- [x] `tailwind.config.ts` / `tailwind.config.js` → colour and spacing tokens
- [x] CSS custom properties from `globals.css` or equivalent → semantic tokens
- [x] `tokens.json` → raw token values if present
- [x] Storybook `*.stories.tsx` files → `argTypes` prop extraction for HOC-wrapped components

**What Studio generates from the scan:**
- [x] DS component catalogue populated in `design_systems.components` JSONB (name, props, variants)
- [x] Token system seeded from the library's token definitions (merged or replaced)
- [x] `ExternalComponent` nodes wired to the correct import path
- [x] Compiler outputs `import { Button } from '@company/ds'` with correct prop + variant usage

**HOC-wrapped component handling:**
- [x] **Layer 1 -- Static analysis** (ts-morph): resolves direct exports and most generic wrappers
- [x] **Layer 2 -- Storybook stories**: `argTypes` extraction for HOC-wrapped components
- [x] **Layer 4 -- Manual fallback**: "Needs manual mapping" flag in UI; user adds props via form before import

> **Deferred:** Layer 3 (runtime Node.js sandbox for propTypes introspection) — not needed for most modern TypeScript libraries.

**Key files:** `apps/web/app/api/studio/import/codebase/route.ts` (new), `apps/web/components/studio/import-codebase-modal.tsx` (new), `apps/web/lib/studio/codebase-scanner.ts` (new), `apps/web/lib/studio/types.ts`, `compiler/emitters/nextjs.ts`

## Known Limitations

- **HOC prop resolution** -- the 4-layer scanner (static analysis → Storybook → runtime → manual fallback) handles most cases, but deeply nested or dynamically constructed HOC chains may still require manual mapping. *Permanent limitation of static analysis; manual fallback is the safety net. Coverage improves as Storybook adoption increases in target codebases.*
- **npm package introspection requires Node.js serverless** -- not available in the Vercel Edge Runtime. *Addressed by routing codebase import through Node.js serverless functions rather than edge functions. No architectural change needed -- just correct routing configuration.*
- **Token sync filesystem watcher is local dev only** -- auto-sync when codebase token files change is not available in Vercel/Supabase cloud deployments. *Permanent constraint for cloud deployments -- file system watching requires a persistent process. Cloud mode uses on-demand sync via the "Sync with codebase" action. This is clearly indicated in the DS panel UI.*
- **Figma Variables import requires a Figma PAT** -- secondary import path only; not required for primary brownfield or greenfield flows. *Permanent constraint of the Figma API -- no workaround. Documented clearly in the import UI.*
- **Multi-theme not emitted by static HTML emitter** -- the HTML/CSS emitter outputs only the default theme. *Addressed in v0.11.0 -- the HTML emitter will emit all theme blocks as separate CSS classes alongside the default `:root` block, with a JS snippet for theme switching. Static (no-JS) mode emits only the default theme as a permanent constraint.*
- **Component variants `cva()` pattern not supported in Expo emitter** -- React Native uses a different conditional style model. *Addressed in v0.11.0 alongside the FRAMEWORK_CONSTRAINTS implementation -- Expo emitter uses `StyleSheet.create()` with conditional spread patterns instead of `cva()`. Variant behaviour is preserved; only the emit pattern differs.*
