# v0.10.5 — Studio Boilerplate & Starter Themes

**Status:** Complete
**Previous:** [v0.10.4](v0.10.4.md) | **Next:** [v0.10.6](v0.10.6.md)

## Summary

The Studio boilerplate is the primary design system path — the one we recommend to every new user. Up to this release it has been a token set in name only: changing `color.primary` in the DS editor has no visible effect on the canvas because the components in `apps/web/components/ui/` are wired to hardcoded Tailwind classes, not the active token set. The canvas shows the same static shadcn defaults regardless of what the user does in the token editor.

This release fixes that at the root, then builds on top of it. The CSS variable bridge means token edits are live. Starter themes give less experienced users an instant visual starting point — five distinct personalities, all using the same component set. The extended component library adds marketing and layout blocks so the boilerplate covers a real product, not just a form widget collection.

The DS picker is restructured so the Studio boilerplate path is clearly primary, and external component libraries are framed as an optional add-on.

---

## Why This Matters

The original pitch was: *connect your tokens to your components and design with your real brand*. Without the CSS variable bridge that pitch is hollow — the canvas is blind to token changes. With it, changing your brand accent from purple to teal updates every Button, Badge, and Link on the canvas instantly. That's the moment the product clicks for a new user.

The starter themes give designers who aren't sure where to start a set of pre-made personalities to pick from. They land in the editor with a complete, consistent visual system already applied — not an empty token list. More experienced users bypass this and define tokens from scratch.

### 0.10.5.6 — Preview Token Bridge

The CSS variable bridge introduced in 0.10.5.1 scopes variables to `[data-canvas-root]`, which only exists in the editor canvas. When Preview is clicked, the screen loads inside an `<iframe>` with no editor context — so all shadcn variables (`--primary`, `--background`, etc.) fell back to browser defaults and DS colours were invisible.

- [x] Extract token → CSS utility (`lib/studio/token-to-css.ts`) with `rawTokensToFlatVars`, `buildShadcnAliases`, and `buildTokenCSS(raw, scope?)` — shared by both the canvas bridge and the preview bridge
- [x] `CanvasTokenBridge` updated to import from the shared utility (no behaviour change)
- [x] `editor-layout.tsx`: both Preview `router.push` calls now include `dsId` in the query string
- [x] `preview/page.tsx`: reads `dsId` from search params; fetches DS tokens from `/api/studio/design-systems/${dsId}`; builds CSS with `:root` scope; sends to the iframe via `postMessage({ type: "studio:previewTokens", css })` after iframe load
- [x] `app/layout.tsx`: inline `<script>` listens for `studio:previewTokens` postMessage and injects/updates a `<style id="studio-preview-tokens">` tag — zero impact on normal page loads

**Key files:** `apps/web/lib/studio/token-to-css.ts` (new), `apps/web/components/studio/canvas-token-bridge.tsx`, `apps/web/components/studio/editor-layout.tsx`, `apps/web/app/preview/page.tsx`, `apps/web/app/layout.tsx`

## Design Decisions

**Why CSS variables over a React context/theme provider?** CSS variables are the simplest mechanism that works across all component sources — Studio's own components, shadcn reference copies, and (eventually) user-owned files via the local module server. A theme provider would require all components to be written to consume it. CSS variables work by convention and require no component-level changes beyond pointing Tailwind config at them.

**Why emit both Studio-namespaced and shadcn-compatible variable aliases?** Studio-namespaced vars (`--color-accent`) are clean and forward-compatible. shadcn-compatible vars (`--primary`) are needed immediately so the 44 existing `components/ui/` files respond to token changes without modification. Both are scoped to `[data-canvas-root]` so they never pollute the editor chrome.

**Why `inheritsComponents` instead of duplicating the components block?** The Studio boilerplate has 40+ components with full metadata and defaultTemplates — duplicating that into 4 new theme files would create 4 separate sources of truth to keep in sync. The API-layer merge approach keeps each theme file to just its token values (~50 lines) while the component definitions live in one place.

**Why remove shadcn from the top-level picker?** Users who pick "shadcn/ui" currently get a token set that matches shadcn defaults and a virtual import path. That's now the Studio boilerplate — the Minimal theme *is* essentially the shadcn visual language. Surfacing it as a separate choice creates confusion ("what's the difference between Studio Minimal and shadcn?"). The answer is: nothing meaningful for a new user.
