# v0.10.5 -- AI Developer Bridge (MCP)

**Status:** Planned
**Previous:** [v0.10.0](v0.10.0.md) | **Next:** [v0.11.0](v0.11.0.md)

## Summary

Exposes Studio UI's design state as an MCP (Model Context Protocol) server so AI coding agents like Claude Code and Cursor can read screen specs, design tokens, component definitions, and change history -- then make intelligent code changes that reflect actual design intent.

This release is dramatically more powerful when built on top of v0.10.0's Design System Authoring. When a brownfield component library is connected, the MCP server already understands the real component API -- variant prop types, slot names, token values -- and can pass that context directly to an AI agent.

The result is the workflow from the post: designer makes a visual change in Studio UI, AI agent in Cursor reads the change via MCP and writes the correct implementation code using the team's actual components.

See [COLLAB_WIRING.md](../COLLAB_WIRING.md) for background on the collaboration and real-time infrastructure.

- [ ] `studio_compile(name)` -- trigger compilation and return generated code
- [ ] `studio_get_design_tokens()` -- fetch resolved token values (all themes)
- [ ] `studio_lint(name)` -- run structure linter and return warnings
- [ ] `studio_get_components()` -- list available components with their prop interfaces

- [ ] Server reads from the same `spec/` directory and API routes as the editor
- [ ] Works in both filesystem mode (local dev) and Supabase mode

**Key files:** `packages/studio-mcp/index.ts` (new), `packages/studio-mcp/package.json` (new), `packages/studio-mcp/resources.ts` (new), `packages/studio-mcp/tools.ts` (new)

## Test Plan

### Prerequisites
- Dev server running
- v0.10.0 complete (at least one DS connected or boilerplate applied)
- Claude Code or MCP Inspector available for end-to-end tests
- A screen with content (e.g., Dashboard)

### Tests

**1. MCP Server Startup**
1. Run `npx studio-mcp` from the project root
2. Verify server starts without errors
3. Connect via MCP Inspector
4. List resources -- verify `studio://screens`, `studio://tokens`, `studio://components`, `studio://changelog` are available

**2. Resource Access**
1. Fetch `studio://screens` -- verify screen names and routes are listed
2. Fetch `studio://screens/Dashboard` -- verify full JSON spec tree is returned
3. Fetch `studio://tokens` -- verify design token values including all themes
4. Fetch `studio://components` -- verify component list (built-in + any connected library components)

**3. Tool Invocation**
1. Call `studio_list_screens()` -- verify names match project screens
2. Call `studio_get_screen_spec("Dashboard")` -- verify spec JSON matches editor state
3. Call `studio_compile("Dashboard")` -- verify compiled code is returned
4. Call `studio_lint("Dashboard")` -- verify linter result (warnings or clean)
5. Call `studio_update_screen_spec("Dashboard", modifiedSpec)` -- verify Studio canvas updates

**4. Design Change Capture**
1. In Studio, add a Button node to a screen and save
2. Fetch `studio://changelog` -- verify entry reads "Added Button to [parent]"
3. Modify the Button's text property
4. Check changelog -- verify "Updated Button text from '...' to '...'"
5. Delete the Button -- verify "Removed Button from [parent]"

**5. Bidirectional Sync**
1. Use Claude Code to call `studio_update_screen_spec` adding a new Heading node
2. Verify Studio canvas shows the new Heading without manual refresh
3. In Studio, modify the Heading text and save
4. Call `studio_get_screen_spec` again -- verify updated text is present

**6. Auto-Discovery**
1. Open Cursor with `.cursor/mcp.json` in the project root
2. Verify Studio UI MCP server appears in Cursor's MCP server list
3. Verify Claude Code picks up `.claude/mcp.json`

**7. Filesystem Mode (No Supabase)**
1. Remove Supabase env vars, restart dev server and MCP server
2. Verify all tools and resources work using filesystem-based specs
3. Verify no errors about missing Supabase connection

## Known Limitations

- MCP server uses stdio transport only (no HTTP/SSE in v0.10.5; can be added in v1.2.0)
- Change log is memory-based (last 100 entries); persistent storage requires filesystem write access or a Supabase table
- `studio_update_screen_spec` performs basic node-type validation but not full semantic checking
- No per-screen permissions or approval gates on MCP writes (deferred to v1.2.0 Enterprise)
- MCP server must be restarted if the spec directory moves (no dynamic watcher in v0.10.5)
- Brownfield component prop introspection quality depends on TypeScript strictness of the source library
